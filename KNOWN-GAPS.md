## AI / connector effector layer (opt-in, off the deploy critical path)

The connector framework is an agent **effector layer** (invoke/scope/approval + budgeted autonomy + delegation-ID + decision-class). It is fully unit/integration tested, but the following are validated against test doubles rather than the live world:

- **Connectors tested against stubs, not live external APIs** (severity: info — the largest current gap). The catalogue is **65 providers** — 49 static-credential + a 12-provider OAuth2 tier + a 4-provider complex-auth tier (SigV4 + checksum; see the bullets below). The static-credential set is the original 4 + 38 commercial-lane providers built 2026-06-13 + 3 Netcore + 4 stragglers (Calendly, SurveyMonkey, Qualtrics, Metabase): WhatsApp, Shopify, Stripe, HubSpot, Zendesk, SendGrid, Twilio, Telegram, Teams/Google-Chat/Zapier/Make/n8n webhooks, Gupshup/Plivo/Exotel/Kaleyra/Sinch, Cashfree, Freshdesk, Intercom, Pipedrive, Freshsales, Mailchimp, WooCommerce, Typeform, Jotform, ServiceNow, Jira, monday, Asana, ClickUp, Trello, Notion, Airtable, ActiveCampaign, Klaviyo, Mailgun, Netcore Email/WhatsApp/SMS, Calendly, SurveyMonkey, Qualtrics, Metabase. Every provider has a unit test, but each is implemented against the **documented** API shape (endpoint, auth header, body) and stubbed at `Net::HTTP`. On 2026-06-13 the 35 real-API providers were **doc-verified** (an adversarial pass checking base/path/method/auth/body against each vendor's current official docs — 29 clean, 2 minor, 4 corrected: pipedrive `/api` path, sinch region host, gupshup `/sm`→`/wa` EOL migration, shopify fulfillment-order flow). But **none has been exercised against a live endpoint with real credentials** — doc-verified ≠ live-tested. Validate each against the live API (and re-check time-sensitive details, e.g. WhatsApp pricing, Shopify REST→GraphQL, monday API-Version) before relying on it in production.
- **OAuth2 tier: 12 providers built, none live-validated** (severity: info). The OAuth2 authorization-code + transparent-refresh seam shipped 2026-06-13 (`Connectors::OauthProvider` + admin connect/callback, commit `6e9c903`). The tier now has **12 providers**: Google Calendar (reference), Gmail, Google Sheets, Google Drive, Microsoft 365 (Outlook mail + calendar), Salesforce, Zoho CRM, HubSpot (OAuth), Dynamics 365, DocuSign, Box, Dropbox. Each is unit-tested with the token exchange/refresh and each action's wire call stubbed at `Net::HTTP` — same **doc-shaped, not live-validated** caveat as the static-credential set (no real OAuth app has been connected end to end; per-provider wrinkles worth live-checking: Salesforce/Zoho persist a per-account API host from the token response, Zoho uses a `Zoho-oauthtoken` header, Dynamics needs a resource-scoped `.default` grant). Remaining roadmap OAuth connectors not yet built: Microsoft Teams / SharePoint / OneDrive, Zoom, Marketo, Adobe Acrobat Sign, Google Ads / LinkedIn, etc. Gov DPI connectors remain licence-gated (see the connector-lane decision).
- **Netcore SMS endpoint is an unverified working assumption** (severity: info). Netcore Email + WhatsApp are built to confirmed doc contracts (Email retries `Authorization: x-api-key` → `bearer` at runtime to settle the documented header disagreement). Netcore SMS uses the CPaaS-Deflector multi-channel send as a working assumption — the standalone-SMS host / method / field-names could not be extracted from Netcore's client-rendered docs, and the auth header (`api-key` → `api_key`) is tried both ways. Settle with one authenticated live call (see `plan/netcore-research-2026-06-13.md`) before production use; the DLT template/sender requirements are confirmed.
- **Complex-auth tier: 4 providers built, signing verified, live contract owed** (severity: info). A non-Bearer auth tier: **Amazon SES** + **Amazon S3** (AWS SigV4, `Connectors::AwsSigv4`) and **PayU** + **PhonePe** (India payment-gateway checksums). The *signing* is verified, not just doc-shaped: SigV4 reproduces AWS's published "get-vanilla" test vector exactly, and the PayU SHA-512 / PhonePe SHA-256 (`X-VERIFY`) checksums are recomputed independently in tests. What's still owed (as for the rest of the catalogue) is one authenticated **live** call per provider to confirm the endpoint/request contract end to end.
- **Complex-auth tier: 4 providers deliberately deferred pending live sandbox credentials** (severity: info, by decision). **Paytm** + **CCAvenue** (proprietary AES-128-CBC checksum / full-payload encryption) and **Snowflake** (key-pair RS256 JWT) + **BigQuery** (service-account JWT → token exchange) were assessed and NOT shipped. Their auth crypto cannot be verified correct without a live sandbox 200 — and shipping unverifiable crypto would violate the doc-verify-before-live bar (a "looks right but fails live" provider is worse than none). Build each against its sandbox when operator credentials are available, anchoring on a known vector or a live round-trip the way SES/PayU were.
- **Agent dispatch loop run only with the `fake` LLM client** (severity: info). `Connectors::AgentRunner` + `Llm::HttpClient#chat_with_tools` are driven by `Llm::FakeClient` in tests; the end-to-end "tool-use turn → `Connectors::Invoke.call`" loop has not been run against a real model endpoint.
- **Decision-class doctrine is judicially untested for AI in India** (severity: info). The autonomous / confirm / of_record gating is grounded in administrative-law principles (reasoned-order duty, non-fettering, audi alteram partem), not in case law applying them to AI-influenced decisions — monitor (e.g. SC Draft AI Regs for Courts 2026).
